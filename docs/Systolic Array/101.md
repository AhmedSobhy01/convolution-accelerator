## Motiviation
The 101 approach was initially adopted due to its **simplicity and ease of implementation**. Each processing element (PE) performs an independent multiplication, and the overall result is obtained by summing all partial products at the array output.
We wanted a *baseline* -like we call it- approach that will sure work smoothly in both data processing (CU & SA) and data getting (AGU & SRAM)

---
## Description
Each PE computes a single multiplication between an input activation and a kernel value. 
Inputs propagate vertically through the array, while kernel values propagate horizontally. 

>[!NOTE]
 Unlike more advanced systolic designs, accumulation is not performed inside the array; instead, each PE outputs a partial product that is collected and reduced in a separate global accumulation stage.

---
## Data Flow

Each PE receives:
- An input value from the top
- A kernel value from the top

Each PE outputs:
- A partial product (`kernel × input`)
- A forwarded input value to the PE below
- A forwarded kernel value to the PE on the below

At the systolic array level:
- Inputs are injected from the top row
- Kernels are injected from the left column
- All partial products produced by the array are forwarded to a centralized summation block

This approach imposes no strict temporal alignment constraints on the inputs or kernels beyond correct spatial placement. But We couldn't find a way to be able to get input data while loading the kernel, so we have to wait for the Kernel to load first.

## Performane Matrices

For an NxN times N×N array:
- Each cycle, the array computes up to N2N^2N2 independent multiplications.
- However, all partial products must be accumulated by a centralized reduction stage to produce a final result.

The reduction requires combining N2N^2N2 values, either through a large combinational adder tree or a multi-cycle sequential accumulator. As a result:
- Output latency increases with array size.
- The maximum operating frequency is limited by the reduction logic.
- The effective throughput of final outputs does not scale proportionally with the number of PEs.

While the 101 approach exposes massive parallelism in multiplication, its reliance on centralized accumulation limits both scalability and sustained throughput.
